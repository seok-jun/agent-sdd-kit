# 프로젝트 적응 가이드

첫 실행에서는 저장소의 구조를 Skill에 맞추지 말고 Skill의 탐색 기준을 저장소에 맞춘다.

## 확인할 항목

| 항목 | 확인할 내용 |
|---|---|
| 비교 기준 | PR base, 원격 기본 브랜치, working tree 포함 여부 |
| 문서 위치 | business 문서 glob과 이름 규칙 |
| 진입점 | HTTP, message, batch, command, retry의 코드 표현 |
| 호출 계층 | 진입점에서 데이터 또는 외부 호출까지의 실제 구조 |
| 문서 인덱스 | 제목, frontmatter, 파일명 중 코드와 연결하는 기준 |
| 백로그 | 선택 경로, 작업 ID, 변경 파일을 기록하는 섹션 |
| 검증 | 실행 가능한 정적 검사, 테스트, 빌드 명령 |

## 결정 우선순위

1. 현재 요청에서 사용자가 명시한 값
2. 가장 가까운 `AGENTS.md` 또는 프로젝트 지침
3. 저장소에서 반복되는 기존 관례
4. 사용자 확인

관례가 하나뿐이거나 서로 충돌하면 규칙으로 단정하지 않는다.

## 진입점 탐색

프레임워크 이름보다 역할을 기준으로 찾는다.

- 외부 요청이나 이벤트를 받는가?
- 실제 비즈니스 심볼을 호출하는가?
- 독립적으로 실행 가능한 경계인가?
- 재시도나 복구 경로가 별도로 존재하는가?

## 호출 체인 종료 지점

다음 중 현재 동작을 설명하기에 필요한 지점까지 읽는다.

- 데이터 조회 또는 저장
- 외부 API 또는 SDK 호출
- 메시지 발행
- 파일 또는 캐시 변경
- 응답 조합과 후처리
- 실패 처리와 재시도

특정 레이어 이름이 없다는 이유로 탐색을 멈추지 않는다. 반대로 프로젝트에 존재하지 않는 DAO나 Mapper 계층을 억지로 찾지 않는다.

## 비교 범위 오류 방지

- base와 HEAD를 항상 출력한다.
- PR diff와 working tree 변경을 구분한다.
- 변경 파일이 예상보다 지나치게 많으면 잘못된 base 가능성을 먼저 확인한다.
- 기본 브랜치 이름을 추측하지 않는다.
- untracked 문서는 되돌릴 안전망이 약하므로 저장 전 확인 단계를 생략하지 않는다.

## Skill 본문을 바꿔야 하는 경우

한 프로젝트의 경로나 클래스명을 Skill에 직접 넣지 않는다. 반복해서 필요한 프로젝트 규칙은 해당 저장소의 `AGENTS.md` 또는 별도 프로젝트 규칙에 둔다.

공개 Skill 자체를 바꾸는 경우는 다음으로 제한한다.

- 여러 저장소에 공통으로 적용되는 절차 오류
- 중지 조건이나 사용자 확인 게이트 누락
- 특정 도구에 종속되지 않는 탐색·검증 개선

# Business 문서 규칙

business 문서는 설계 의도가 아니라 코드의 현재 동작을 짧게 설명한다. 코드가 최종 기준이며 문서는 다음 분석을 빠르게 시작하기 위한 캐시다.

## 문서 위치

위치를 Skill이 고정하지 않는다. 다음 우선순위로 결정한다.

1. 프로젝트의 `AGENTS.md`, `CLAUDE.md` 또는 문서 규칙
2. 이미 존재하는 business 문서의 관례
3. 사용자가 지정한 경로

셋 모두 없으면 경로를 제안하고 사용자 확인을 받는다.

## 문서 단위

기본 단위는 외부에서 동작을 시작시키는 진입점 하나다.

- HTTP endpoint
- message topic 또는 consumer
- scheduled job 또는 batch command
- retry 또는 recovery entrypoint

같은 업무명이라도 동작 경로가 다르면 별도 문서로 둔다. 여러 진입점이 하나의 흐름을 이루어 분리하면 이해할 수 없는 경우에만 묶고 사용자 확인을 받는다.

## 최소 골격

프로젝트에 기존 형식이 없으면 다음을 사용한다.

```markdown
# {기능명} ({진입점 식별 정보})

## 확인 필요
{근거를 확인하지 못한 항목. 없으면 섹션을 생략한다.}

## 흐름
{호출 순서, 분기, 외부 호출, 데이터 접근}

## 주의
{잘못 변경하기 쉬운 예외와 공유 동작. 없으면 "없음"}
```

제목에는 문서를 코드 진입점과 연결할 수 있는 식별 정보가 있어야 한다.

## 포함 기준

다음 질문에 `예`라면 포함한다.

> 이 정보를 모르면 다음 작업자가 현재 동작을 잘못 변경할 가능성이 있는가?

주로 포함하는 내용:

- 호출 순서와 분기 조건
- 외부 계약과 주요 파라미터 매핑
- 데이터 저장·조회 조건과 식별 키
- 실패 무시, 후처리, 재시도, 부분 성공
- 다른 진입점이 공유하는 서비스나 데이터
- 코드가 존재하지만 현재 실행되지 않는 조건

주로 제외하는 내용:

- 디렉터리 구조 전체
- 문서 도입 배경과 회고
- 코드와 중복되는 긴 구현 설명
- 근거 없이 추정한 정책과 설계 의도

## 갱신 정책

### 부분 갱신이 기본

- 기존 문서 전체를 읽는다.
- 변경이 닿은 문장과 인접한 맥락만 코드와 다시 대조한다.
- 확인하지 않은 기존 문장을 임의로 정리하거나 삭제하지 않는다.
- 새로 쓰거나 바꾼 문장에는 검토용 코드 근거를 붙인다.

### 전체 재작성

다음 중 하나에 해당할 때만 제안한다.

- 문서의 진입점이나 문서 단위가 현재 코드와 맞지 않는다.
- 기존 설명이 여러 곳에서 코드와 충돌한다.
- 호출 체인 자체가 대폭 교체됐다.
- 사용자가 전체 기준선 재확인을 요청했다.

전체 재작성 전에는 이유, 읽을 코드 범위, 잃을 수 있는 기존 정보를 보여주고 사용자 확인을 받는다. 재작성할 때는 유지할 모든 문장도 코드에서 근거를 다시 확인한다.

## SDD 문서와의 관계

- SDD 문서는 이번 변경의 의도와 작업 이력이다.
- business 문서는 구현 후 코드의 현재 동작이다.
- SDD 내용을 복사해 business 문서를 만들지 않는다.
- 백로그는 영향 범위를 대조하는 보조 자료일 뿐 코드 근거를 대체하지 않는다.

## 확인 용어

- `확인`: 사용자가 대상이나 저장을 명시적으로 승인한 상태
- `검토 초안`: 채팅에 표시되며 아직 파일에 쓰지 않은 내용
- `코드 근거`: 현재 비교 범위의 실제 파일과 위치로 다시 찾을 수 있는 정보

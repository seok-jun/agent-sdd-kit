# SDD 문서 뼈대 생성

빈 골격 문서를 단계별로 만들어 이후 분석과 설계의 틀을 고정한다. 이 Skill은 코드 조사나 실제 내용 작성을 수행하지 않는다.

등급 판정은 [references/task-grading.md](references/task-grading.md)를 따른다.

골격 생성 뒤의 조사와 서술은 별도 Skill을 전제하지 않는다. 같은 에이전트 또는 후속 세션이 일반 작업으로 수행할 수 있다. 사람은 작업 등급, 생성 범위, Stage 완료를 확인하는 게이트 역할을 한다.

## 절대 규칙

1. **확정 전에는 파일을 만들지 않는다.** 현재 Stage에서 생성할 파일 목록, 경로, 섹션 헤더, 주요 표 헤더, 문서 관계를 먼저 출력하고 사용자 확인 후에만 저장한다. 사용자가 일부 항목만 확인했거나 전체 제안에 대한 승인인지 모호하면 다시 확인한다.
2. **확인받은 대상 외에는 생성하거나 수정하지 않는다.** 이번 실행에서 기록한 변경 경로를 확인 목록과 대조한다. 이를 위해 저장소 전체 파일 목록이나 git 상태를 조사하지 않는다. 확인된 생성 예정 경로에 파일이 이미 있으면 덮어쓰지 말고 보고한 뒤 재사용 또는 수정 범위를 다시 확인받는다.
3. **SDD 문서끼리는 반드시 연결한다.** 모든 문서는 제목 바로 아래에 문서 관계 표를 둔다. 아직 생성되지 않은 문서는 파일명만 적고 링크하지 않는다. 실제로 생성된 시점에만 상대 링크로 바꾼다. 같은 Stage에서 함께 생성되는 문서는 링크를 허용한다.
4. **입력값과 분석 내용을 보충하기 위해 조사하지 않는다.** 코드, 기존 문서, 저장소 파일 목록, git 상태와 로그를 읽어 값을 추측하지 않는다. 사용자가 제공하지 않은 값은 묻는다. 사용자 확인 후에는 확인된 생성 예정 경로의 존재 여부와 이번 Stage에서 생성하거나 수정한 파일의 링크·생성 범위만 검증할 수 있다. 사용자가 구조 검증을 요청하거나 허용한 경우에는 선언된 SDD 문서 경로에 한해 기존 Stage 파일을 읽어 파일 존재와 링크 상태를 확인할 수 있다. 사용자가 링크·상태 갱신 대상으로 확인한 기존 SDD 문서는 해당 수정에 필요한 범위로 읽을 수 있다.
5. **TODO 문구는 상수다.** 아래 템플릿의 `> TODO:` 뒤 문장을 한 글자도 바꾸지 않고 복사한다. 상황에 맞게 다듬거나 알아낸 사실을 덧붙이거나 요약하지 않는다. 표 셀과 목록의 `TODO`도 실제 내용으로 바꾸지 않는다.
6. **Stage를 건너뛰지 않는다.** Stage 완료는 파일 존재나 TODO 교체 여부가 아니라 사용자의 명시적인 확인으로만 판정한다. 이전 Stage의 완료 확인 전에는 다음 Stage 파일을 제안하거나 만들지 않는다.

규칙 4와 5는 뼈대 생성 과정에서 검증되지 않은 사실이 다음 세션으로 넘어가는 것을 막기 위한 것이다.

## 역할과 완료 기준

| 구간       | 에이전트                                    | 사람                       |
| -------- | --------------------------------------- | ------------------------ |
| 작업 시작    | 사용자 설명만으로 등급 후보를 제안한다.                  | 등급을 확정한다.                |
| Stage 골격 | 파일 목록, 구조, 관계를 제안하고 확인된 골격만 생성한다.       | 생성 범위를 확정한다.             |
| 내용 작성    | 이 Skill 밖의 일반 작업으로 코드 조사, 분석, 설계를 수행한다. | 결과를 검토하고 Stage 완료를 확인한다. |
| 다음 Stage | 이전 Stage 완료 확인 후 다음 골격을 제안한다.           | 진행 여부를 결정한다.             |

Stage 3에 진입했다는 것은 Stage 1의 as-is 분석과 Stage 2의 to-be 설계 및 change plan이 작성되고 사용자에게 완료 확인을 받았다는 뜻이다.

## 입력으로 확인할 것

먼저 작업 등급을 확인한다. 등급이 없으면 코드나 저장소를 조사하지 않고 사용자가 설명한 작업 범위만으로 후보를 제안한 뒤 확인받는다.

### Trivial 또는 Small

SDD 전용 입력은 요구하지 않는다. 최소 기록에 필요한 내용만 확인한다.

### Medium 이상

- **작업 등급** - Trivial / Small / Medium / Large / Epic
- **feature 슬러그** - 폴더명에 사용할 kebab-case 문자열
- **작업 ID prefix** - Stage 2부터 사용할 대문자 식별자
- **관련 코드 범위** - 문서 관계 표에 적을 사용자 제공 경로 또는 설명
- **기준 ref** - as-is를 고정할 commit hash, tag 또는 branch
- **작성일** - 현재 날짜 또는 사용자가 지정한 날짜

관련 코드 범위와 기준 ref의 실재 여부를 뼈대 생성 단계에서 확인하지 않는다.

## Trivial과 Small의 최소 기록

`Trivial` 또는 `Small`이면 SDD 파일을 만들지 않는다.

기록 위치는 다음 순서로 정한다.

1. 사용자가 지정한 Issue, Ticket 또는 PR 본문
2. 이미 사용하는 작업 관리 문서
3. 별도 위치가 없으면 채팅 출력

최소 기록을 위해 새 Markdown 파일을 만들지 않는다.

### Trivial

```markdown
## Trivial 작업 기록

- 변경 이유: {사용자가 제공한 내용 또는 확인 필요}
- diff 확인: {확인 기준 또는 확인 필요}
```

### Small

```markdown
## Small 작업 기록

- 변경 목적: {사용자가 제공한 내용 또는 확인 필요}
- 수정 대상: {사용자가 제공한 내용 또는 확인 필요}
- 수정 금지 영역: {사용자가 제공한 내용 또는 확인 필요}
- 검증 기준: {사용자가 제공한 내용 또는 확인 필요}
```

프롬프트에 없는 값은 조사하지 않고 `확인 필요`로 남기거나 사용자에게 묻는다.

## Stage 재진입

이 Skill은 Stage마다 별도 세션에서 다시 호출될 수 있다.

재진입 시 다음 두 값을 먼저 확인한다.

- 현재 진행할 Stage
- 이전 Stage가 완료됐다는 사용자의 명시적인 확인

Stage 2 재진입이면 초기 골격 생성 전인지, TO-BE 매핑 후 개별 change plan 생성 단계인지와 사용자가 확인한 작업 ID도 확인한다.

사용자의 진술을 완료 판정의 기준으로 삼는다. 파일이 존재하거나 TODO가 채워졌다는 사실만으로 완료를 추측하지 않는다.

`선언된 SDD 문서 경로`는 사용자가 직접 제공했거나 이전 생성 제안에서 확인받은 정확한 파일 경로를 뜻한다. 경로를 새로 발견하기 위한 디렉터리 목록 조사는 하지 않는다.

사용자가 구조 검증을 요청하거나 허용한 경우에만 선언된 SDD 문서를 읽어 파일 존재와 링크 상태를 확인한다. 이때도 코드나 git을 읽어 분석 내용을 보충하지 않는다. (규칙 4의 예외 적용)

## 문서 관계 규칙

모든 문서는 제목 바로 아래에 다음 표를 둔다.

```markdown
| 항목 | 값 |
|---|---|
| 문서 종류 | {파일별 고정값} |
| feature | {feature} |
| 작업 ID | {작업 ID 또는 해당 없음} |
| 선행 문서 | {파일별 고정값} |
| 후속 문서 | {생성된 문서 링크 또는 미정} |
| 관련 코드 | {사용자가 제공한 코드 범위} |
```

as-is 문서에만 다음 두 행을 추가한다.

```markdown
| 기준 ref | {기준 ref} |
| 작성일 | {작성일} |
```

파일별 선행 관계는 다음과 같다.

| 파일                              | 선행 문서                                                               |
| ------------------------------- | ------------------------------------------------------------------- |
| `README.md`                     | 없음                                                                  |
| `01-as-is-flow.md`              | `[README.md](./README.md)`                                          |
| `02-to-be-mapping.md`           | `[01-as-is-flow.md](./01-as-is-flow.md)`                            |
| `change-plans/README.md`        | `[02-to-be-mapping.md](../02-to-be-mapping.md)`                     |
| `change-plans/{PREFIX}-{NN}.md` | `[02-to-be-mapping.md](../02-to-be-mapping.md)`                     |
| `backlog/{PREFIX}-{NN}.md`      | `[change-plans/{PREFIX}-{NN}.md](../change-plans/{PREFIX}-{NN}.md)` |
| `implementation-backlog.md`     | `[02-to-be-mapping.md](./02-to-be-mapping.md)`                      |

후속 문서는 실제 파일이 생기기 전까지 `미정`으로 둔다. 파일 생성 시 선행 문서의 후속 문서와 README의 문서 구성 표도 확인 목록에 포함해 갱신한다.

같은 변경 항목은 to-be 매핑, change plan, backlog에서 동일한 작업 ID를 사용한다. 작업 ID는 에이전트가 임의로 만들지 않고 사용자가 확인한 항목에만 부여한다.

**표기 고정:** 제목 구분자는 ASCII 하이픈 `-`, 흐름 화살표는 `->`를 사용한다. em dash와 유니코드 화살표를 사용하지 않는다.

## 생성 단계

한 번에 전체 문서를 만들지 않는다. 각 Stage에서 골격을 생성한 뒤 멈추고, 실제 내용 작성과 사용자 완료 확인을 기다린다.

### Stage 1 - 현행 파악

다음 두 파일만 제안한다.

```text
docs/{feature}-sdd/
  README.md
  01-as-is-flow.md
```

골격 생성 후 일반 에이전트 작업으로 관련 코드를 조사하고 `01-as-is-flow.md`의 실제 내용을 작성한다.

사용자가 as-is 분석 완료를 명시적으로 확인하기 전에는 Stage 2를 시작하지 않는다.

### Stage 2 - 목표 정의

먼저 다음 두 파일을 제안한다.

```text
docs/{feature}-sdd/
  02-to-be-mapping.md
  change-plans/
    README.md
```

골격 생성 후 일반 에이전트 작업으로 TO-BE 매핑을 작성한다. `변경 필요` 또는 `신규 필요`로 확인된 항목마다 작업 ID를 부여한다.

개별 change plan은 같은 Stage에 재진입해 확정된 작업 ID에 대해서만 제안하고 생성한다.

```text
docs/{feature}-sdd/
  change-plans/
    {PREFIX}-{NN}.md
```

사용자가 TO-BE 매핑과 모든 change plan의 설계 완료를 명시적으로 확인하기 전에는 Stage 3을 시작하지 않는다.

### Stage 3 - 실행 추적

Stage 3에 진입하면 분석과 설계는 완료된 상태다. 확정된 작업 ID에 대해서만 다음 파일을 제안한다.

```text
docs/{feature}-sdd/
  backlog/
    {PREFIX}-{NN}.md
  implementation-backlog.md
```

골격 생성 후 일반 에이전트 작업으로 확정된 change plan을 구현 가능한 Todo와 선행 조건으로 분해한다. 사용자가 실행 백로그 완료를 확인하면 SDD 준비 단계를 종료한다.

## 파일 템플릿

### README.md

```markdown
# {feature} SDD

| 항목 | 값 |
|---|---|
| 문서 종류 | README |
| feature | {feature} |
| 작업 ID | 해당 없음 |
| 선행 문서 | 없음 |
| 후속 문서 | [01-as-is-flow.md](./01-as-is-flow.md) |
| 관련 코드 | {사용자가 제공한 코드 범위} |

## 목적
> TODO: 이 SDD가 고정하려는 것 한 문단.

## 기준 자료
> TODO: as-is 코드 진입점, TO-BE 명세 위치, 불명확 항목 확인 방법.

## 문서 구성
| 문서 | 내용 | 상태 |
|---|---|---|
| [01-as-is-flow.md](./01-as-is-flow.md) | 현행 흐름 | 작성 중 |
| 02-to-be-mapping.md | 목표 매핑 | 미생성 |

## 작업 원칙
> TODO: 유지할 스택/제약, 우선 사용할 기존 구조, 변경 금지 대상.
```

README의 문서 구성 표는 파일이 실제로 생성될 때마다 갱신한다. 미생성 문서에는 링크를 걸지 않는다.

문서 구성 표의 상태값은 TODO 상수가 아니라 Stage 진행에 따라 갱신하는 관리값이다.

- 파일이 없으면 `미생성`
- 골격을 만들었지만 실제 내용 작성과 사용자 완료 확인이 끝나지 않았으면 `작성 중`
- 실제 내용 작성 후 사용자가 Stage 완료를 확인했으면 `완료`

### 01-as-is-flow.md

```markdown
# {feature} - 현행 흐름

| 항목 | 값 |
|---|---|
| 문서 종류 | as-is |
| feature | {feature} |
| 작업 ID | 해당 없음 |
| 선행 문서 | [README.md](./README.md) |
| 후속 문서 | 미정 |
| 관련 코드 | {사용자가 제공한 코드 범위} |
| 기준 ref | {기준 ref} |
| 작성일 | {작성일} |

## 처리 흐름
> TODO: 진입점 -> 비즈니스 처리 -> 데이터 또는 외부 호출 흐름.

## 참고한 business 문서
> TODO: 경로와 기준 ref. 없으면 "없음". 존재를 확인하기 전에는 링크하지 않는다.

## 위험 지점
> TODO: 잘못 변경하면 기존 동작이 깨질 수 있는 지점.

## 확인 못 한 것
> TODO: 코드만으로 판단할 수 없는 항목. 없으면 "없음".
```

### 02-to-be-mapping.md

```markdown
# {feature} - TO-BE 매핑

{to-be 문서 관계 표}

| 작업 ID | 업무 | TO-BE | 현행 | 판정 | 변경 계획 |
|---|---|---|---|---|---|
| TODO | TODO | TODO | TODO | TODO | TODO |

판정 기준: 매칭 / 변경 필요 / 신규 필요 / 확인 필요
```

작업 ID와 change plan 링크는 항목이 확정되고 파일이 생성된 뒤에만 입력한다.

### change-plans/README.md

```markdown
# Change plans

{change plan index 문서 관계 표}

| 작업 ID | 제목 | 상태 | 문서 |
|---|---|---|---|
| TODO | TODO | TODO | TODO |
```

### change-plans/{PREFIX}-{NN}.md

```markdown
# {PREFIX}-{NN}. {제목}

{change plan 문서 관계 표}

## 변경 대상
> TODO: 수정 / 참고 / 변경 금지 파일을 구분한다.

## 변경 내용
```

### backlog/{PREFIX}-{NN}.md

```markdown
# {PREFIX}-{NN} Backlog

{backlog 문서 관계 표}

## Todo
- [ ] TODO

## Blocked
- 없음

## 변경 파일
- 대기

## Done
- 대기
```

`## 변경 파일`은 뼈대 단계에서는 `대기`로 둔다. 구현하면서 저장소 루트 기준 경로를 기록한다. 이 목록은 후속 business 문서 작성의 조사 시작점이며 전체 호출 체인 목록으로 간주하지 않는다.

### implementation-backlog.md

```markdown
# 구현 백로그

{통합 backlog 문서 관계 표}

| 작업 ID | 기능 | 우선순위 | 상태 | 선행 조건 |
|---|---|---|---|---|
| TODO | TODO | TODO | TODO | TODO |
```

## 실행 절차

1. 작업 설명만으로 등급 후보를 제안하고 사용자 확인을 받는다.
2. `Trivial` 또는 `Small`이면 최소 기록을 출력하고 파일을 만들지 않은 채 멈춘다.
3. `Medium` 이상이면 현재 Stage와 이전 Stage 완료 확인 여부를 확인한다.
4. 현재 Stage에 필요한 입력값을 확인한다. 없는 값은 조사하지 않고 묻는다.
5. 생성 예정 파일, 경로, 섹션 헤더, 주요 표 헤더, 문서 관계, 함께 갱신할 기존 SDD 문서를 출력한다.
6. 사용자 확인을 받는다.
7. 확인된 생성 예정 경로의 존재 여부를 확인한다. 이미 존재하면 덮어쓰지 않고 보고한다. 존재하지 않는 파일만 템플릿 그대로 생성하고, 확인된 기존 SDD 문서만 링크와 상태 갱신을 위해 수정한다.
8. 이번 실행에서 생성하거나 수정한 파일만 읽어 상대 링크와 생성 범위를 검증한다.
9. 실제 내용 작성에 필요한 후속 작업과 현재 Stage의 완료 조건을 알리고 멈춘다.

## 출력 형식

```markdown
## 입력값
{등급, feature, ID prefix, 관련 코드 범위, 기준 ref, 작성일}

## 현재 Stage
{Stage와 이전 Stage 완료 확인 여부}

## 생성 예정 파일
{파일 목록}

## 갱신 예정 파일
{링크와 상태 갱신이 필요한 기존 SDD 문서 또는 없음}

## 문서별 구조
{파일명, 섹션 헤더, 주요 표 헤더}

## 문서 관계
{선행/후속 연결}

## 확인 필요
{사용자가 결정해야 할 값}
```

## 금지

- 사용자 확인 전 파일 생성 또는 수정
- 확인 목록 밖의 파일 생성 또는 수정
- 템플릿 TODO를 실제 분석이나 설계 내용으로 채우기
- 코드, git, 기존 문서를 조사해 입력값이나 분석 내용 보충하기
- 존재하지 않는 파일에 링크하기
- 작업 ID를 임의로 만들기
- 사용자 완료 확인 없이 다음 Stage로 진행하기

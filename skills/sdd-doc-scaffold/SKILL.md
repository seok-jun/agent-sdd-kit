# SDD 문서 골격 생성

SDD 문서의 빈 골격을 단계별로 만든다. 사실 조사와 설계 내용 작성은 후속 작업으로 남긴다.

등급 기준은 [references/task-grading.md](references/task-grading.md), 문서별 골격은 [references/document-templates.md](references/document-templates.md)를 따른다.

이 Skill과 함께 배포된 `references/` 파일은 실행 지침의 일부다. 필요한 reference를 읽는 것은 프로젝트 조사로 보지 않으며 별도 사용자 확인을 요구하지 않는다.

## 절대 규칙

1. 파일을 만들기 전에 생성 목록, 경로, 섹션 헤더와 주요 표 헤더, 문서 관계를 출력하고 사용자 확인을 받는다.
2. 확인받은 목록 밖의 파일은 만들거나 수정하지 않는다.
3. 입력값이나 분석 내용을 보충할 목적으로 코드, 기존 파일, 파일 목록, git 상태와 로그를 조사하지 않는다. 사용자 확인 후에는 이번 Stage에서 생성한 파일에 한해 존재 여부, 링크, 생성 범위를 검증할 수 있다.
4. 사용자가 주지 않은 값은 추측하거나 저장소에서 찾지 말고 질문한다.
5. 템플릿의 치환 자리와 TODO 문구를 실제 내용처럼 채우지 않는다.
6. Stage 완료는 파일 내용이나 존재 여부가 아니라 사용자의 명시적인 확인으로만 판정한다. 현재 Stage의 완료 확인 전에는 다음 Stage 파일을 만들지 않는다.

## 입력

먼저 작업 등급을 확인한다.

`Trivial` 또는 `Small`이면 feature 슬러그, 작업 ID prefix, 기준 ref, 작성일 같은 SDD 전용 입력을 요구하지 않는다. 아래 최소 기록에 필요한 값만 확인한다.

`Medium` 이상이면 다음 값을 확인한다.

- 작업 등급: `Trivial`, `Small`, `Medium`, `Large`, `Epic`
- feature 슬러그: SDD 폴더명에 사용할 kebab-case 문자열
- 작업 ID prefix: Stage 2부터 사용할 대문자 식별자
- 관련 코드 범위: 사용자가 제공한 경로 또는 설명
- 기준 ref: as-is를 고정할 commit hash, tag 또는 branch
- 작성일: 현재 날짜 또는 사용자가 지정한 날짜

작업 등급이 없으면 작업 설명만으로 등급 후보를 제안할 수 있다. 코드를 조사해 판정하지 말고 사용자 확인을 받는다.

`Trivial` 또는 `Small`이면 SDD 파일을 만들지 않는다. 대신 아래 최소 기록을 채팅에 출력하고, 사용자가 지정한 Issue, Ticket, PR 본문 또는 기존 작업 관리 문서에 옮길 수 있게 한다. 기록 위치가 없으면 채팅 출력으로 끝내며 새 파일을 만들지 않는다.

### Trivial 최소 기록

```markdown
## Trivial 작업 기록

- 변경 이유: {사용자가 제공한 내용 또는 확인 필요}
- diff 확인: {확인 기준 또는 확인 필요}
```

### Small 최소 기록

```markdown
## Small 작업 기록

- 변경 목적: {사용자가 제공한 내용 또는 확인 필요}
- 수정 대상: {사용자가 제공한 내용 또는 확인 필요}
- 수정 금지 영역: {사용자가 제공한 내용 또는 확인 필요}
- 검증 기준: {사용자가 제공한 내용 또는 확인 필요}
```

위 기록을 채우기 위해 코드를 조사하지 않는다. 프롬프트에 없는 값은 `확인 필요`로 남기거나 사용자에게 묻는다.

## Stage 재진입

이 Skill은 Stage마다 별도 세션에서 다시 호출될 수 있다.

재진입 시 현재 Stage와 이전 Stage의 사용자 확인 여부를 먼저 묻는다. 사용자의 명시적인 진술을 기준으로 진행한다.

사용자가 구조 검증을 요청하거나 허용한 경우에만 선언된 SDD 문서를 읽어 파일 존재와 링크 상태를 확인한다. 파일이 존재하거나 TODO가 채워졌다는 사실만으로 Stage가 확인됐다고 판단하지 않는다.

재진입 과정에서도 코드나 git을 읽어 입력값 또는 분석 내용을 보충하지 않는다.

## Stage 1 - 현행 파악

다음 두 파일만 제안한다.

```text
docs/{feature}-sdd/
  README.md
  01-as-is-flow.md
```

`01-as-is-flow.md`에 대한 사용자의 완료 확인 전에는 Stage 2를 시작하지 않는다.

## Stage 2 - 목표 정의

다음 파일을 제안한다.

```text
docs/{feature}-sdd/
  02-to-be-mapping.md
  change-plans/
    README.md
```

to-be 매핑에서 실제 변경 항목이 확인된 뒤 각 항목에 작업 ID를 부여한다. ID가 확정되지 않은 change plan 파일은 만들지 않는다.

## Stage 3 - 실행 추적

다음 파일을 제안한다.

```text
docs/{feature}-sdd/
  backlog/
    {PREFIX}-{NN}.md
  implementation-backlog.md
```

backlog 파일은 확정된 작업 ID에 대해서만 만든다.

## 문서 관계

- 모든 문서는 제목 바로 아래에 문서 관계 표를 둔다.
- 아직 만들어지지 않은 문서는 파일명만 적고 링크하지 않는다.
- 파일이 실제로 생성된 시점에만 상대 링크로 바꾼다.
- 같은 변경 항목은 to-be 매핑, change plan, backlog에서 동일한 작업 ID를 사용한다.

## 실행 절차

1. 입력값과 작업 등급을 확인한다. 재진입이면 현재 Stage와 이전 Stage의 사용자 확인 여부도 묻는다.
2. `Trivial` 또는 `Small`이면 최소 기록을 출력하고 파일을 만들지 않은 채 멈춘다.
3. `Medium` 이상이면 현재 Stage에서 만들 파일과 각 파일의 섹션 헤더·주요 표 헤더를 출력한다.
4. 문서 관계의 선행·후속 연결을 출력한다.
5. 사용자 확인을 받는다.
6. 확인받은 파일만 템플릿으로 만든다.
7. 이번 Stage에서 생성한 파일만 읽어 상대 링크와 생성 범위를 검증한다.
8. 이번 실행에서 자신이 생성·수정한 경로 기록을 확인 목록과 대조한다. 이 대조를 위해 git 상태나 저장소 전체 파일 목록을 조사하지 않는다.
9. 후속 분석·설계 작업과 다음 Stage 재호출 조건을 알리고 멈춘다.

## 출력 형식

```markdown
## 입력값
{등급, feature, ID prefix, 관련 코드 범위, 기준 ref, 작성일}

## 생성 예정 파일
{파일 목록}

## 문서별 구조
{파일명, 섹션 헤더, 주요 표 헤더}

## 문서 관계
{선행·후속 연결}

## 확인 필요
{사용자가 결정해야 할 값}
```

## 금지

- 확인 전 파일 생성
- 템플릿을 실제 분석 내용으로 채우기
- 코드, git, 기존 문서를 조사해 입력값 보충하기
- 존재하지 않는 파일에 링크하기
- 작업 ID를 임의로 만들기
- 현재 Stage를 건너뛰기
